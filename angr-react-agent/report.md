# ReAct Agent + angr Reverse-Analysis Report

## Experiment Objective

The objective is to use a ReAct-style agent to inspect the supplied crackme,
guide angr toward the success behavior, avoid the failure and dead-loop paths,
and solve the required input.

## Environment and Dependencies

- Python 3.10 or 3.11 is recommended.
- GCC builds the target with `gcc crackme.c -o crackme`.
- Python dependencies are `angr`, `openai`, and `langchain-core`.
- Install them with `pip install -r requirements.txt`.

No API key is required for the reproducible deterministic run. If an existing
`OPENAI_API_KEY` is provided, the optional LLM integration can word the
planning thoughts. Tool observations always come from angr.

## Tool Definitions

- `inspect_binary(binary_path=None) -> dict`: loads the binary, runs CFGFast,
  reports architecture and entry address, lists useful functions, and locates
  the success, wrong-password, and dead-loop strings.
- `explore_success_path(binary_path=None) -> dict`: creates symbolic stdin,
  finds a state whose stdout contains `Success! Flag is found.`, and avoids
  states printing either failure or trap text.
- `solve_input(state_id) -> dict`: retrieves the saved success state and asks
  its solver for concrete password bytes.

## ReAct Loop

The orchestrator records three complete Thought -> Action -> Observation
rounds. Round 1 gathers semantic evidence, round 2 converts that evidence into
symbolic-execution find/avoid goals, and round 3 concretizes the successful
symbolic input. The observations in `logs/run.txt` are generated by executing
the implemented tools, not handwritten placeholders.

## Logged Rounds Summary

1. Inspect the binary to identify success, failure, and trap semantics.
2. Explore toward success while avoiding wrong-password and dead-loop output.
3. Solve the symbolic stdin retained by the successful state.

## Result

Final solved input: `AZcE`

The third byte satisfies `input[2] ^ 0x12 == 'q'`, and the fourth satisfies
`input[3] + 3 == 'H'`.

## Role of the LLM

The LLM is the orchestration and semantic-planning layer. angr is the formal
symbolic-execution and constraint-solving layer. The LLM uses high-level
semantic cues such as `Success`, `Wrong password`, `trapped`, and `dead loop`
to select find/avoid goals and reduce unnecessary path exploration. This
reduces search-space difficulty by steering angr away from irrelevant failure
branches and the non-terminating trap instead of exploring every path equally.
